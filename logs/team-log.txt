- **Before:** 1,676 lines with `DuckDBManager` singleton, disk-backed connections, double `get_readonly_connection()` bug, MV stubs, column cache
- **After:** 707 lines — pure in-memory DuckDB, context-managed connections, all views auto-created

  - Removed `ensure_materialized_views()` and all MV stubs

- **`services/versioned_cache.py`:** Replaced `get_etl_version()` to use `admin/etl_state.json` instead of SQLite MV metadata

### Zero SQLite Guarantee
- All SQLite references removed from active codebase
- SQLite queue (`etl_queue.sqlite`) remains for IPC only (admin → scheduler communication), not for queries

### Problem
Inventory page caused SIGKILL due to excessive memory consumption:
- `update_exec_summary` callback opened 3 DuckDB connections (3×4GB=12GB) + 2 Polars scans
- 7 `.apply(lambda row: ..., axis=1)` calls caused slow row-by-row iteration
- `_get_snapshot_date()` had connection leak (no context manager)
- Dead code (`_query_stock_levels`, `_get_snapshot_date`) still in file

### Changes Made

**`services/inventory_metrics.py` — Full rewrite (877→742 lines)**
- ❌ Removed `_get_snapshot_date()` — dead code with connection leak risk
- ❌ Removed `_query_stock_levels()` — dead code (only in legacy_ops)
- ✅ Replaced all 5 Polars scans with DuckDB SQL queries
- ✅ Consolidated multi-query functions into single `_query_conn()` contexts:
  - `get_stock_levels_ledger()`: Polars + 2×DuckDB → 1×DuckDB
  - `get_sell_through_analysis()`: 2×DuckDB + Polars → 1×DuckDB
  - `_query_abc_products()`: DuckDB + Polars → 1×DuckDB
  - `get_inventory_costs()`: 2×Polars → 1×DuckDB
  - `query_inventory_summary()`: 2×DuckDB → 1×DuckDB
- ✅ Vectorized all `.apply(axis=1)` with numpy:
  - `_query_location_ledger_deltas`: `df.apply(calculate_delta)` → SQL CASE WHEN
  - `get_stock_levels_ledger`: vectorized days_of_cover, flags

**`pages/inventory.py`**
- Removed unused `query_inventory_summary` import
